To Monitoring & Logging HTTPS File Uploads on Client-Side Linux Machines

The Core Challenge: Encryption's Blind Spot

The very encryption that protects user data in transit also creates a paradox for security teams. HTTPS, especially with Perfect Forward Secrecy (PFS), renders traditional network monitoring ineffective, creating a "blind spot" that adversaries exploit for data exfiltration and malware delivery. A Multi-Layered Solution Strategy

No single tool provides complete visibility. A robust strategy requires combining different approaches. Explore the three primary layers of monitoring below to understand the tools, techniques, and trade-offs for each.

Network-Level Interception

This approach involves decrypting HTTPS traffic to inspect its contents. While providing deep visibility, it comes with significant technical, performance, and legal challenges. It's often implemented using a Man-in-the-Middle (MITM) proxy.

Pros

  • Deep visibility into payload content.
  • Can inspect for malware and sensitive data patterns.
  • Works for any application using standard HTTPS.

Cons

  • Requires trusting a custom CA certificate on all clients.
  • Can be defeated by certificate pinning.
  • High performance overhead.
  • Major privacy and legal concerns.

mitmproxy is a powerful open-source tool for intercepting and inspecting HTTPS traffic. Below is an example Python addon script to log file upload details.


import logging
import hashlib
import re
from mitmproxy import http

logging.basicConfig(
    filename='/var/log/mitmproxy_uploads.log',
    level=logging.INFO,
    format='%(asctime)s - %(message)s'
)

class FileUploadLogger:
    def request(self, flow: http.HTTPFlow) -> None:
        if flow.request.method in ["POST", "PUT"]:
            if flow.request.multipart_form:
                logging.info(f"Multipart upload to {flow.request.pretty_url}")
                for name, value in flow.request.multipart_form.items():
                    file_hash = hashlib.sha256(value).hexdigest()
                    logging.info(f"  Part: {name.decode()}, Size: {len(value)} bytes, SHA256: {file_hash}")
            elif flow.request.content:
                content_type = flow.request.headers.get("Content-Type", "unknown")
                if "x-www-form-urlencoded" not in content_type:
                    file_hash = hashlib.sha256(flow.request.content).hexdigest()
                    filename = "unknown_file"
                    if "Content-Disposition" in flow.request.headers:
                        match = re.search(r'filename="([^"]+)"', flow.request.headers["Content-Disposition"])
                        if match:
                            filename = match.group(1)
                    logging.info(f"Direct upload to {flow.request.pretty_url}")
                    logging.info(f"  File: {filename}, Type: {content_type}, Size: {len(flow.request.content)} bytes, SHA256: {file_hash}")

addons = [FileUploadLogger()]

Host-Level Auditing

This approach focuses on monitoring activity directly on the Linux kernel. It provides the ground truth of what files are being accessed and what network connections are being made by which processes, bypassing the complexities of encryption entirely.

The Linux Auditing System (`auditd`) is a native kernel-level tool. It can be configured to log specific system calls related to file access and network connections.


# Monitor file writes in user home and temp directories
-w /home/ -p wa -k user_file_write
-a always,exit -F arch=b64 -S openat,creat -F dir=/tmp/ -k temp_file_activity

# Monitor network connection attempts
-a always,exit -F arch=b64 -S connect -k network_connect

Correlate events by Process ID (PID) and timestamp in /var/log/audit/audit.log to link a file access to a network connection.

eBPF provides a modern, low-overhead way to achieve kernel observability. Tools like Falco use eBPF to detect suspicious behavior in real-time.


# Example Falco Rule
- rule: "Suspicious File Write on Endpoint"
  desc: "Detects file writes in user home or temp dirs"
  condition: >
    (open_write or creat) and 
    (fd.name startswith /home/ or fd.name startswith /tmp/) and 
    not proc.name in (known_safe_processes)
  output: "Suspicious file write (user=%user.name proc=%proc.name file=%fd.name)"
  priority: WARNING

Endpoint & Application Insights

This layer provides the highest-fidelity information by leveraging solutions that operate directly on the endpoint with content and application awareness. This is often the most effective approach for corporate environments.

Endpoint Data Loss Prevention (DLP)

DLP agents on the client machine inspect file content *before* it gets encrypted and sent. They can enforce policies to block unauthorized uploads of sensitive data (e.g., PII, source code) to cloud services and provide detailed logs of what file was sent where, by whom.

Browser-Level DLP Connectors

For enterprise environments, browsers like Google Chrome offer direct integration with third-party DLP vendors. This allows the browser to send the full content and metadata of uploaded files to a DLP agent for analysis, providing deep, application-aware visibility without unstable extensions.

Endpoint Detection & Response (EDR)

EDR solutions provide comprehensive endpoint visibility by monitoring processes, file system activity, and network connections. They use behavioral analysis to detect data exfiltration patterns, such as a process reading many sensitive files and then making an unusual outbound connection.
